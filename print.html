Cross-Platform Title Screen

Introduction

Cross-Platform Compatability is a great tool for developers and players. It allows you to play your favorite games with all your friends regardless of the device. Developing a game to be compatible with all platforms is difficult; One of the hardest parts is creating UI that works with all devices.

Today, we'll create a title screen that's compatible with computers, consoles, and mobile devices. Before you start, here's some vocabulary that we'll be discussing, and what it means.

I decided to make this lesson written instead of a video/live session, mostly because it's a lot less work on my part, but it also allows people to go through at their own pace, instead of watching a video.

Prerequisites

  • Basic understanding of Luau as a language
  • Basic understanding of Roblox Instances, and their properties
  • Basic understanding of Events, Functions, and Connections

Vocabulary

  • Scale: The size/position of an object in %. When not parented to a GUI object, it will take up a % of the screen. When parented to a GUI object, it will take up a % of the parent's size. If a GuiObject is set to {0.5, 0, 0.5, 0} when parented to a GUI object, it will take up 50% of the parent's size, horizontally and vertically

Represented by: {ScaleX, OffsetX, ScaleY, OffsetY}

  • Offset: The size/position of an object in pixels. Does not take into account the size of the parent. If a GuiObject is set to {0, 150, 0, 150} when parented to a GUI object, it will take up 150 pixels horizontally and vertically.

Represented by: {ScaleX, OffsetX, ScaleY, OffsetY}

Basic Design

Before we can script the UI, we'll need to properly design the menu before anything else.

To start, we'll want to create a ScreenGui to display GUI. After that, we'll create a frame named "Holder", to hold our UI elements. This allows us to scale our parent frame, and resize the children at the same time. Before we start adding children to the Holder frame, we'll want to edit a few properties.

  • Edit the AnchorPoint of the Holder frame to 0.5, 0.5 so the scaling is at the center.
  • Edit the Size of the Holder frame to 1, 0, 1, 0 so that the scaling takes up the entire screen.
  • Edit the Position of the Holder frame to 1, 0, 1, 0 so that the position
  • Finally, change the Transparency of the Holder frame to 1 so that the frame appears invisible.
  • Set BorderSizePixel to 0.

Preparation for scripting

We'll change the Size and Position to scale, alongside changing the AnchorPoint to 0.5, 0.5. Once that's done, we can place the buttons anywhere we want, and we're almost ready to start scripting. In order to take advantage of certain cross-platform compatability tools, we'll need to change certain properties in our buttons.

To make sure the text is readable on all devices, we'll want to turn the TextScaled property to true, so it's not too big or small on any device.

Next, we'll going to change certain properties to help make the UI compatible with consoles. For the Play button, we'll want to change the SelectionObjectDown to the Credits button, so users on controllers can select the UI by simply going down on the control pad. We'll want to do the same process for the Credits button, but change it to SelectionObjectUp.

After that, we'll want to set the Selectable property to true on both of our TextButtons. If you aren't familiar with this event, here's a quick rundown of what it does and how we can use it.

The selectable property decides whether controller users can navigate to that GUI object when navigating UI with a controller and/or WASD keys. When set to true, the UIObject becomes navigatable.

Then we want to set the Active property to true., The Active property in GuiButtons (or in this specific case, TextButtons) decides whether that UIButton will detect inputs or not. When the property is set to true, it opens up the ability for us to connect a function to GuiButton.Activated, a cross-platform event that allows us to detect inputs from controllers, mobile, and mouse/keyboard users.

Scripting The Menu

To start scripting our UI, we'll want to insert a LocalScript into StarterGui. Name this local script "UI_Manager". This is where we'll be doing the main scripting of our UI. 

-------------------------------------------------------
-- Services
local PlayerService = game:GetService("Players") -- so we can get our local player 
local GuiService = game:GetService("GuiService") -- so we can add controller & WASD/Arrow key support

Above, we define two of our crucial services. PlayerService, which allows us to access the LocalPlayer (whoever is playing on their computer), and GuiService, a service that allows us to add xbox compatability to our UI, alongside navigate the menus with text.

Next, we'll want to define some player-related values so we can get the player's copy of the UI, and manipulate that.

-------------------------------------------------------
-- Player Objects
local Player = PlayerService.LocalPlayer
local Character = Player.Character or Player.CharacterAdded:Wait()
local PlayerGui = Player.PlayerGui
local Controls = require(Player.PlayerScripts:WaitForChild("PlayerModule")):GetControls() -- we can use this to disable/enable the player's controls.

When getting a player's UI, it's important to use PlayerGui instead of StarterGui, because of how replication works. Every client gets their own copy of whatever is in StarterGui, and instead of having a single player controlling everyone else's GUI, we can manipulate just the client's GUI by using PlayerGui.

Ok great, we have our player objects down, and we have a way to access our GUI, so the next step is to define the GUI! Below we define our GUI objects so we can use them in code.

-------------------------------------------------------
-- UI Objects || Core
local ScreenGui		 = PlayerGui:WaitForChild("ScreenGui")
local Holder		 = ScreenGui:WaitForChild("Holder")


-------------------------------------------------------
-- Frames || Buttons
local CreditsFrame	 = Holder.CreditsFrame
local PlayButton	 = Holder.Play
local CreditsButton  = Holder.Credits
local BackButton 	 = Holder.CreditsFrame.Back

Nothing advanced here, we're just getting references to our objects so we can use them. We're almost done! All we need to do now is define some visal effects, alongside a table to make later sections a bit easier to script and manage. 

-------------------------------------------------------
-- UI Effects || ETCs
local HoverColor	 = Color3.fromRGB(89, 183, 255)
local OriginalColor  = Color3.fromRGB(255, 255, 255)
local ButtonTable    = {PlayButton, CreditsButton}
local Connections	 = {}

Why are we storing buttons in ButtonTable? Simple, we want to store them in the table so we can use a for loop, instead of having to manualy connect to certain events one by one.

Now that we've gotten all our examples down, we can start writing functions to connect to our events.

-------------------------------------------------------
-- Functions
function PlayButtonActivated()
	ScreenGui.Enabled = false
	GuiService.SelectedObject = nil
	Controls:Enable()
	
	for i, connection in pairs(Connections) do
		connection:Disconnect()
	end
end

function CreditsButtonActivated()
	CreditsButton.Visible = false
	PlayButton.Visible = false
	
	CreditsFrame.Visible = true
	GuiService.SelectedObject = BackButton
end

function BackButtonActivated()
	CreditsFrame.Visible = false
	
	CreditsButton.Visible = true
	PlayButton.Visible = true
	GuiService.SelectedObject = PlayButton
end

Pretty simple stuff, right? A lot of the code adjusts button/frame visibility, and PlayButtonActivated() disables the ScreenGui when the play button is pressed. GuiService.SelectedObject allows us to set a selected object for controller users, alongside enabling WASD/Arrow Key menu navigation.

-------------------------------------------------------
-- Connections
PlayButton.Activated:Connect(PlayButtonActivated)
BackButton.Activated:Connect(BackButtonActivated)
CreditsButton.Activated:Connect(CreditsButtonActivated)

for index, button in pairs(ButtonTable) do
	local Enter1 = button.MouseEnter:Connect(function()
		button.BackgroundColor3 = HoverColor
	end)
	
	local Leave1 = button.MouseLeave:Connect(function()
		button.BackgroundColor3 = OriginalColor
	end)
	
	local Enter2 = button.SelectionGained:Connect(function()
		button.BackgroundColor3 = HoverColor
	end)
	
	local Leave2 = button.SelectionLost:Connect(function()
		button.BackgroundColor3 = OriginalColor
	end)
	table.insert(Connections, Enter1)
	table.insert(Connections, Leave1)
	table.insert(Connections, Enter2)
	table.insert(Connections, Leave2)
end

Controls:Disable()
GuiService.SelectedObject = PlayButton

We're almost done, all we have to do now is connect all our events. All GuiButtons have an event called Activated, which allows us to detect mouse button 1 clicks, touch inputs, and people confirming inputs on controller.

We also connect our MouseEnter and MouseLeave events, so we can detect when the user hovers over an object. Alongside that, we connect our SelectionGained and SelectionLost events, which allow us to detect whenever a GuiObject gains/loses selection.

Finally, we insert our connected events to a table, for later use. It's important to store your events somewhere, so they can be disconnected when they are no longer needed. This saves memory, and improves game performance.
